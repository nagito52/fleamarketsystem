# Fairsty 開発振り返り（就活提出用）

このドキュメントは、`README.md` の補足として「なぜこの設計にしたか」「開発を通して何を学んだか」をまとめたものです。  
要件定義は別途作成済みで、本資料では実装・改善・今後の方針に焦点を当てます。

## 1. プロジェクトの狙い

- フリーマーケットの基本体験（出品・購入・取引管理）を一気通貫で実装する
- 認証/認可、決済、外部通知、管理機能まで含めた実務寄りの構成を作る
- 単なる CRUD ではなく、状態遷移と権限制御を重視した業務ロジックを実装する

## 2. 実装で特に注力した点

### 2-1. 認証・権限・セッション

- Spring Security で一般ユーザーと管理者のロールを分離
- 新規登録直後の自動ログイン不整合を修正し、セッションに認証状態を正しく反映
- 管理画面の無効化状態がログイン可否に反映されるように修正

### 2-2. 取引フローの整合性

- 取引ステータス（出品中 -> 取引中 -> 発送済 -> 売却済）を基準に操作を制御
- 発送通知後は購入者がキャンセルできない仕様を実装
- 管理者の強制キャンセルは「取引中」のみ実行可能に制限

### 2-3. 外部サービス連携

- Stripe API で決済・返金フローを実装
- Cloudinary で画像アップロード・削除を管理
- LINE Messaging API で通知を送信（認証エラー時のログを改善）

### 2-4. フロントエンド体験

- Vanilla JS とカスタム CSS で UI を再設計
- 購買意欲と親しみやすさを意識したカラー設計（コーラル & ティール）
- Cropper.js により出品画像のトリミング機能を追加

## 3. 学んだこと

### 3-1. 設計と実装の往復

- 要件からエンティティ設計へ落とし込む際、初期設計の精度が開発速度に大きく影響する
- 仕様変更に追従できるよう、サービス層に責務を集約する設計の重要性を再認識

### 3-2. 不具合対応の実践力

- 静的リソース配信、認証セッション、DBスキーマ不足など、実運用で起きやすい問題を経験
- 原因調査 -> 再現 -> 修正 -> 再検証の流れを短いサイクルで回せるようになった

### 3-3. 可読性と保守性

- 未使用フィールドや未使用 import の削除を継続し、コード品質を改善
- エラーログを具体化し、障害時に追跡しやすい状態へ改善

## 4. 反省点と改善方針

### 4-1. 反省点

- テーブル追加などスキーマ変更が途中で発生し、初期設計の不足が露出
- テストコードが十分でなく、回帰確認を手動に依存する場面があった
- 一部セキュリティ設定（暗号化方式など）は改善余地がある

### 4-2. 改善方針

- 要件定義段階で ER 図・状態遷移図・権限制御表を明確化
- サービス層の単体テストとコントローラーの統合テストを優先実装
- 本番を想定したセキュリティ設定（例: BCrypt など）へ段階的に移行

## 5. 今後のロードマップ

### 短期（1-3ヶ月）

- テスト整備（サービス層の単体テスト、主要機能の統合テスト）
- 例外ハンドリング統一（`@ControllerAdvice` の導入）
- クエリ最適化とページネーションの導入

### 中期（3-6ヶ月）

- 検索機能の強化（絞り込み条件の拡張）
- 通知機能の拡張（メール・Push 連携）
- 監視と運用基盤（ログ可視化、CI/CD）を整備

### 長期（6ヶ月以上）

- API 指向の構成へ段階的に拡張
- コンテナ化を含むデプロイ自動化
- 不正検知やレコメンドなどの付加価値機能を検討

## 6. 使用技術（README準拠）

- Backend: Java 17, Spring Boot, Spring Security, Spring Data JPA, Thymeleaf
- Frontend: HTML, CSS, JavaScript（Vanilla JS）, Cropper.js
- Database: PostgreSQL
- Build: Maven
- External Services: Stripe API, Cloudinary, LINE Messaging API
